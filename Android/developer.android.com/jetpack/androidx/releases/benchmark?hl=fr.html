Déclarer des dépendances

Pour ajouter une dépendance à Benchmark, vous devez ajouter le dépôt Maven de Google à votre projet. Pour en savoir plus, consultez la section Dépôt Maven de Google.

Macrobenchmark

Pour utiliser Macrobenchmark dans votre projet, ajoutez les dépendances suivantes à votre fichier build.gradle pour votre module Macrobenchmark :

Groovy

dependencies {
  androidTestImplementation "androidx.benchmark:benchmark-macro-junit4:1.4.1"
}

Kotlin

dependencies {
  androidTestImplementation("androidx.benchmark:benchmark-macro-junit4:1.4.1")
}

Microbenchmark

Pour utiliser Microbenchmark dans votre projet, ajoutez les dépendances suivantes au fichier build.gradle de votre module Microbenchmark :

Groovy

dependencies {
    androidTestImplementation "androidx.benchmark:benchmark-junit4:1.4.1"
}

android {
    ...
    defaultConfig {
        ...
        testInstrumentationRunner "androidx.benchmark.junit4.AndroidBenchmarkRunner"
    }
}

Kotlin

dependencies {
    androidTestImplementation("androidx.benchmark:benchmark-junit4:1.4.1")
}

android {
    ...
    defaultConfig {
        ...
        testInstrumentationRunner = "androidx.benchmark.junit4.AndroidBenchmarkRunner"
    }
}

The Microbenchmark library also provides a Gradle plugin to use with your microbenchmark module. This plugin sets build configuration defaults for the module, sets up benchmark output copy to the host, and provides the ./gradlew lockClocks task.

To use the plugin, include the following line in the `plugins` block in your top-level build.gradle file:

Groovy

plugins {
  id 'androidx.benchmark' version '1.4.1' apply false
}

Kotlin

plugins {
  id("androidx.benchmark") version "1.4.1" apply false
}

Then apply the plugin to your benchmark module's build.gradle file

Groovy

plugins {
  id 'androidx.benchmark'
}

Kotlin

plugins {
    id("androidx.benchmark")
}

Commentaires

Vos commentaires nous aident à améliorer Jetpack. N'hésitez pas à nous contacter si vous découvrez de nouveaux problèmes ou si vous avez des idées pour améliorer cette bibliothèque. Veuillez consulter les problèmes existants de cette bibliothèque avant d'en signaler un nouveau. Vous pouvez ajouter votre vote à un problème existant en cliquant sur le bouton en forme d'étoile.

Version 1.4.1

10 septembre 2025

Publication d'androidx.benchmark:benchmark-*:1.4.1. La version 1.4.1 contient ces commits.

Correction de bugs

  • Correction de perfettoSdkTracing=true (par exemple, lorsque le traçage de Compose est effectué dans un microbenchmark) qui abandonnait le processus cible s'il était déjà en cours d'exécution et qu'aucun StartupMode n'était spécifié. (Ib2c1f)

Version 1.4.0

30 juillet 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0. La version 1.4.0 contient ces commits.

Changements importants depuis la version 1.3.0

Microbenchmark

  • Les tâches Gradle lockClocks et unlockClocks ont été déplacées vers les projets de benchmark, au lieu d'être au niveau supérieur, afin de respecter l'isolation des projets Gradle.
  • BenchmarkRule a été refactorisé pour être basé sur des coroutines et offrir un meilleur comportement yield(). Cela devrait réduire considérablement le risque d'ANR lors des exécutions de benchmarks, en particulier les longues exécutions CI. Remarque : Les benchmarks d'UI doivent s'exécuter avec measureRepeatedOnMainThread.

Macrobenchmark

  • Ajout d'une solution de contournement sur l'API 34 et versions ultérieures pour CompilationMode.None(), qui aurait des performances incohérentes en raison de la compilation partielle des applications par ART après le premier lancement.
  • Fonctionnalité expérimentale : Startup Insights peut mettre en évidence certains problèmes courants dans un Macrobenchmark de démarrage en transmettant MacrobenchmarkRule(..., experimentalConfig = ExperimentalConfig(StartupInsightsConfig(isEnabled = true))).
  • Ajout de ArtMetric, qui peut être utilisé pour détecter la compilation JIT et le chargement de classe non optimisé. Ces deux éléments sont utiles pour valider les optimisations du profil de référence.

Profils de référence

  • BaselineProfileRule collecte désormais les profils pour les applications multiprocessus.

Autres changements

  • TraceProcessor a été extrait dans sa propre bibliothèque (androidx.benchmark:benchmark-traceprocessor) afin de pouvoir être utilisé en dehors des métriques Macrobenchmark, dans d'autres cas. Il peut également être exécuté sur la JVM du bureau en définissant votre propre ServerLifecycleManager.

Version 1.4.0-rc01

18 juin 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0-rc01. La version 1.4.0-rc01 contient ces commits.

Correction de bugs

  • Ajout d'une solution de contournement pour les images d'exécution qui empêchent CompilationMode.None() de mesurer les performances dans le pire des cas après la première itération. Malheureusement, cette solution de contournement nécessite un délai de cinq secondes pour corrompre intentionnellement l'image d'exécution au début de chaque suite de macrobenchmarks (I4a4f1).

Version 1.4.0-beta02

4 juin 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0-beta02. La version 1.4.0-beta02 contient ces commits.

Modifications apportées à l'API

  • Ajout d'un BaselineProfileConfig.Builder pour faciliter l'appel de BaselineProfileRule.collectWithResults() pour les développeurs Java. (I94905)

Version 1.4.0-beta01

7 mai 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0-beta01. La version 1.4.0-beta01 contient ces commits.

Modifications apportées à l'API

  • Ajout d'une variante de constructeur PerfettoTraceRule qui accepte un PerfettoConfig (Ie53ba)

Correction de bugs

  • Mise à jour du format de lien vers les insights de démarrage TraceProcessor pour utiliser le bon plug-in et un délimiteur plus clair (: est plus court que %3A, et les deux sont acceptés) (Ie18ef)
  • Utilisez toujours l'arrêt forcé pour mettre fin aux processus, même lorsque l'appareil est rooté et que vous mettez fin à des applications système. Corrige les exceptions de la forme : Expected no stdout/stderr from killall ... No such process. (Idca2c)

Contribution externe

  • Ajout d'un gestionnaire d'exceptions de proxy pour TraceProcessorHttpServer (I480f5)

Version 1.4.0-alpha11

9 avril 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0-alpha11. La version 1.4.0-alpha11 contient ces commits.

Modifications apportées à l'API

  • Le délai d'attente de l'argument de TraceProcessor a été modifié et est désormais défini sur timeoutMs pour faciliter l'utilisation par l'appelant Java. (I9fbb5)
  • Marquez le constructeur TraceProcessor comme interne. Les appelants doivent utiliser TraceProcessor.startServer ou TraceProcessor.runServer. (Ia8c5b)

Correction de bugs

  • Lorsque vous arrêtez l'application avec MacrobenchmarkScope.killProcess, validez les résultats de la commande d'arrêt pour éviter les échecs silencieux et les erreurs de délai avant expiration. (I84555)

Mises à jour des dépendances

  • Cette bibliothèque cible désormais le langage Kotlin 2.0 et nécessite KGP 2.0.0 ou version ultérieure. (Idb6b5)

Version 1.4.0-alpha10

26 mars 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0-alpha10. La version 1.4.0-alpha10 contient ces commits.

Modifications apportées à l'API

  • Le délai avant expiration par défaut pour la charge et les requêtes du serveur TraceProcessor a été augmenté à 120 secondes (contre 60/30 auparavant). Les deux sont désormais configurables avec un seul paramètre de délai avant expiration. (Ifec87)

Correction de bugs

  • Correction de quelques problèmes qui se produisaient lors de l'analyse comparative ou de la capture de profils d'une application sans profileinstaller et son BroadcastReciever inclus. Cela n'affecte que les exécutions sur les appareils rootés. (Ied308)

Version 1.4.0-alpha09

12 mars 2025

Publication d'androidx.benchmark:benchmark-*:1.4.0-alpha09. La version 1.4.0-alpha09 contient ces commits.

Modifications apportées à l'API

  • Ajout de TraceProcessor et de l'API Session avec des Handles fermables, pour une utilisation plus facile avec des cycles de vie personnalisés. Cela permet également de faciliter l'utilisation des API Coroutine et Java. Les fonctions d'extension de TraceProcessor.runServer {} sont désormais marquées comme expérimentales, car elles sont susceptibles d'être déplacées et de ne plus être expérimentales à l'avenir. (I358b4)

Correction de bugs

  • Correction du problème de capture des profils de référence et de base qui ne fonctionnait pas avec l'API 36 en raison d'une modification de la toybox pgrep qui nécessite désormais -a pour imprimer la ligne de commande complète. (Idc991)
  • Filtrer la configuration de traçage par défaut pour réduire le risque de perte de données dans les traces sur les niveaux d'API plus récents. (I54e8a)
  • Ajout de l'argument d'instrumentation androidx.benchmark.killExistingPerfettoRecordings expérimental, qui peut être défini sur false pour permettre la poursuite de la capture de trace Perfetto préexistante. Par défaut, les captures de trace Perfecto préexistantes sur l'appareil sont arrêtées pour éviter toute interférence. (I02a3c)
  • Le champ JSON context.osCodenameAbbreviated sera désormais REL pour les versions d'OS publiées avec un niveau d'API 35 ou supérieur, car les noms de code non numériques ne sont plus acceptés par la plate-forme sous-jacente. (Ib17fd)
  • Correction d'un plantage dans FrameTimingMetric lorsque des images resynchronisées se produisent. (I7c6f4, b/394610806)
  • Ne partez plus du principe que Choreographer#doFrame est le haut du frame de pile sur le thread principal pour FrameTimingQuery. (Iee0e0, b/340206285)

Version 1.4.0-alpha08

12 février 2025

Version 1.2.0

18 octobre 2023

Publication d'androidx.benchmark:benchmark-*:1.2.0. Liste des commits de la version 1.2.0

Changements importants depuis la version 1.1.0

Profils de référence

  • Le nouveau plug-in Baseline Profile Gradle automatise la capture et l'inclusion des profils de référence dans votre workflow de test et de compilation.
  • BaselineProfileRule.collect est désormais stable. Il s'agit d'une version simplifiée de l'ancienne API expérimentale BaselineProfileRule.collectBaselineProfile.
    • Il vous suffit de spécifier packageName et de piloter votre application.
  • Pour les bibliothèques générant des profils de référence, vous pouvez désormais filtrer les règles générées dans le code (argument BaselineProfileRule.collect) ou encore plus simplement dans le plug-in Gradle.
  • Corrections
    • Correction de la collecte des profils de référence sur Android U+ (Id1392, b/277645214)

Macrobenchmark

  • Compilation
    • Macrobenchmark réinitialise désormais correctement et complètement l'état de compilation pour chaque compilation. Cela nécessite de réinstaller l'APK avant Android 14. Il est donc fortement recommandé d'effectuer des benchmarks sur Android 14 ou version ultérieure si vous souhaitez conserver l'état (comme la connexion utilisateur) dans ce qui est mesuré.
    • Vous pouvez également contourner ce problème en contrôlant la compilation de l'application séparément et en ignorant la compilation avec CompilationMode.Ignore() ou l'argument d'instrumentation.

  • Arguments d'instrumentation

    • Prise en charge de l'argument d'instrumentation androidx.benchmark.dryRunMode.enable (déjà disponible dans microbenchmark) pour des exécutions de validation plus rapides (par exemple, lors de la création du benchmark ou avant l'envoi)
    • Prise en charge de androidx.benchmark.profiling.mode=StackSampling et MethodTracing.
    • Ajout de androidx.benchmark.enabledRules pour permettre le filtrage au moment de l'exécution des tests de règles de profil de référence par rapport aux tests de règles de macrobenchmark
    • Ajout de l'argument androidx.benchmark.perfettoSdkTracing.enable pour activer le traçage avec tracing-perfetto, par exemple le traçage de recomposition Compose. Notez que, lorsqu'il est utilisé avec StartupMode.COLD, le timing sera considérablement affecté, car la bibliothèque de traçage est chargée et activée au démarrage de l'application.

  • Conditions requises

    • Macrobenchmark nécessite désormais ProfileInstaller 1.3.0 ou version ultérieure dans l'application cible pour activer la capture / réinitialisation du profil et l'effacement du cache du nuanceur.

  • Nouvelles API de métriques expérimentales

    • Ajout de l'API expérimentale TraceSectionMetric, qui permet d'extraire un timing simple des blocs trace("") {} de votre application ou TraceMetric pour exploiter toute la capacité de requête de Perfetto TraceProcessor.
    • Ajout de PowerMetric expérimental pour capturer les informations sur la consommation d'énergie
    • Ajout de MemoryCountersMetric expérimental pour comptabiliser les erreurs de page
    • Ajout de l'API expérimentale PerfettoTraceProcessor, qui est utilisée en interne pour extraire des métriques à partir des traces système (également appelées traces Perfetto).

  • Correctifs

    • Résolution des plantages lors de l'installation ou de l'extraction de profils à partir d'une application installée à partir de plusieurs APK (par exemple, à partir d'un app bundle).
    • Correction de FrameTimingMetric qui ignorait les frames avec des ID de frame incohérents (généralement, les frames pendant les ondulations sur l'API 31 et versions ultérieures). (I747d2, b/279088460)
    • Correction des erreurs d'analyse sur les traces de plus de 64 Mo (Ief831, b/269949822)
    • Clarification des erreurs lorsque l'image de l'OS de l'appareil (en particulier de l'émulateur) n'est pas correctement configurée pour le traçage ou la compilation
    • Vérification du niveau de la batterie ignorée pour les appareils sans batterie (micro et macro)
    • Amélioration de la sortie de fichier, avec des erreurs plus claires pour les répertoires de sortie non valides et des valeurs par défaut plus sûres
    • Amélioration de la stabilité de StartupMode.COLD en supprimant systématiquement le cache du nuanceur (également exposé via MacrobenchmarkScope.dropShaderCache)
    • Correction du remplacement Leanback pour startActivityAndWait.

Microbenchmark

  • Fonctionnalités
    • Le profilage a été déplacé dans une phase distincte, après les autres métriques. Ainsi, une exécution de test peut afficher à la fois des résultats précis sur la temporalité et sur le profilage.
  • API expérimentales
    • Ajout de l'API expérimentale MicrobenchmarkConfig pour définir des métriques personnalisées et configurer le traçage et le profilage. Peut être utilisé pour capturer des traces de méthode ou des points de trace (mais attention à la surcharge de traçage).
    • Ajout d'API expérimentales pour contrôler BenchmarkState séparément de BenchmarkRule, sans JUnit
    • Ajout d'un enregistrement PerfettoTrace expérimental pour permettre la capture de traces Perfetto, avec une configuration personnalisée, distincte des API de benchmark.
  • Corrections
    • Solution de contournement pour les espaces blancs manquants en début de ligne dans la sortie du benchmark Android Studio.
    • Correction d'un problème qui pouvait empêcher l'affichage des avertissements dans la sortie du benchmark Android Studio.
    • Correction du plantage SampledProfiling sur Android 13 (API 33) et versions ultérieures.
    • Amélioration considérable des performances de dryRunMode en ignorant le traçage IsolationActivity et Perfetto (mode de simulation jusqu'à 10 fois plus rapide sur les anciennes versions de l'OS).

L'API 29 n'est plus obligatoire, car TraceSectionMetric fonctionne avec androidx.tracing.Trace à des niveaux d'API inférieurs, avec l'utilisation de forceEnableAppTracing dans l'application cible. La bibliothèque de benchmark vous permet d'écrire des benchmarks de performances du code de votre application et d'obtenir des résultats rapidement.

  • Priorisation automatique des threads
  • Prise en charge des tests de performances de l'interface utilisateur, comme dans l'exemple RecyclerView
  • Préchauffage et bouclage prenant en compte JIT
  • Résultats du benchmark au format JSON pour le post-traitement
