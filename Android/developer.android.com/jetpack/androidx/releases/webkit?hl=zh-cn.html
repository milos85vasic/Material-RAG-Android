版本 1.15

版本 1.15.0-alpha02

2025 年 8 月 27 日

发布了 androidx.webkit:webkit:1.15.0-alpha02。版本 1.15.0-alpha02 中包含这些提交内容。

新功能

  • 添加了 Profile#preconnect，允许应用在导航到某个来源之前先打开与该来源的连接，以便加快未来的加载速度。(I8db41)
  • 在 WebViewCompat 中添加了新的 prerenderUrlAsync API，允许应用在网址显示在 WebView 中之前推测性地预渲染网址。
    预渲染可通过提前提取网页来大幅提高加载性能。当用户导航到相应网址时，系统会立即显示预渲染的网页。
    这些 API 之前以实验性 API 的形式发布，现在无需使用 @OptIn 注解即可使用。(Ie14f5)
  • 现在，您可以在调用 startUpWebView API 时指定一组要同步加载的配置文件。这是通过将配置文件名称传递给 WebViewStartUpConfig.Builder 中的新方法 (I9a6e6) 来配置的
  • 向 WebSettingsCompat 添加了一个新 API，用于在 WebSettings 对象上启用/停用 BackForwardCache。此 API 之前以实验性 API 的形式发布，现在无需使用 @OptIn 注解即可使用。(Ie413b)
  • 添加了对替换 User-Agent 设备规格的支持。UserAgentMetadata 类新增了用于获取和设置 User-Agent 元数据设备规格的方法。UserAgentMetadata.Builder 新增了 setFormFactors() 方法，用于替换 Sec-CH-UA-Form-Factors 用户代理客户端提示。您可以通过新的 getFormFactors() 方法检索当前替换值。（I6d08c、b/430554841）

API 变更

  • 向 ProcessGlobalConfig 添加了 API，以配置 WebView 的界面线程初始化（I5e85a、b/433273850、b/397372092）
  • 为 WebViewStartupConfig.Builder#setProfilesToLoadDuringStartup API 添加了新功能名称 (I36451)
  • 将默认 minSdk 从 API 21 移至 API 23（Ibdfca、b/380448311、b/435705964、b/435705223）
  • 更新了来源匹配的标头 API，以允许通过针对同一标头名称多次调用 setOriginMatchedHeader 来覆盖标头值。还添加了一个新的 hasOriginMatchedHeader 方法，用于检查标头名称是否已被使用。
    此变更还将该 API 标记为实验性 API，因为该 API 目前仅支持为每个标头值设置单个值和来源映射，并且未来可能会进行变更，以支持映射到不同来源的多个值。（I03365、b/419469873、b/429269470）
  • 移除了对 Profile#prefetchUrlAsync() 的界面线程要求。现在，可以从任何线程调用它。(I83d57)
  • 已移除 OutcomeReceiverCompat 接口，取而代之的是正式版 androidx.core.os.OutcomeReceiverCompat。该库现在需要 compileSdk 35 或更高版本。(Ie1ea5)
  • 在 WebSettingsCompat 中将 [is/set]IncludeCookiesOnShouldInterceptRequestEnabled 重命名为 [are/set]CookiesIncludedInShouldInterceptRequest。（Ife9e9、b/428924452）
  • 为之前被省略的 Profile.ExperimentalUrlPrefetch 添加了实验性 API 注解。（Ic54eb、b/417458061）

版本 1.15.0-alpha01

2025 年 7 月 2 日

发布了 androidx.webkit:webkit:1.15.0-alpha01。版本 1.15.0-alpha01 中包含这些提交内容。

新功能

  • 一项新 API，用于在向特定来源发送的请求中设置 HTTP 标头名称-值对。
    Profile.setOriginMatchedHeader 方法允许应用指定，当从使用给定配置的 WebView 和服务工作线程发送请求时，应在所有 HTTP/HTTPS 请求中设置标头名称和值，以发送到与提供的许可名单匹配的来源。该 API 仅允许为每个标头名称设置一个映射，如果标头名称已设置，则会抛出异常。
    使用 Profile.clearOriginMatchedHeader 可移除现有映射，使用 Profile.clearAllOriginMatchedHeaders 可移除所有映射。（Ida7f1、b/419469873）
  • 一个新 API，用于在 WebView 和 Service Worker 的 shouldInterceptRequest 回调中访问和设置 Cookie。
    目前在 shouldInterceptRequest 中访问 CookieManager API 的应用开发者现在可以直接通过调用 setIncludeCookiesOnShouldInterceptRequest 启用 Cookie 标头。然后，WebView 会向 WebResourceRequest.getRequestHeaders() 映射添加 Cookie 标头，该标头将是适用于相应请求的确切 Cookie。这还将正确处理任何分区 Cookie。
    应用可以通过新添加的 WebResourceResponseCompat.setCookies API 提供 Set-Cookie 标头值。添加的 Set-Cookie 值将由 WebView 处理，作为响应处理的一部分。（Idbfd1、b/414769380）
  • 一项新的实验性 API，用于预热 Profile 的渲染器进程。
    具有复杂 WebView 生命周期应用可以使用此 API 来确保渲染器进程正在运行，即使所有 WebView 实例都已被销毁。(Ia8fc8)

API 变更

  • 向实验性 Navigation 接口添加了一个新方法，用于提供导航的网址。(I1c6c0)
  • 更新了“个人资料预提取”功能名称，明确指出现在可以在后台线程上调用 Prefetch API。(I7c83a)

版本 1.14.0

2025 年 6 月 4 日

发布了 androidx.webkit:webkit:1.14.0。版本 1.14.0 中包含这些提交内容。

自 1.13.0 以来的重要变更

  • 推出 PaymentRequest API，用于通过 org.chromium.intent.action.PAY intent 调用 Android 原生支付应用。PaymentRequest 默认处于停用状态，WebView 主机应用可以调用 WebSettingsCompat.setPaymentRequestEnabled(settings, true) 来启用它。此外，PaymentRequest 需要在 AndroidManifest.xml 中添加 <queries> 标记。如需了解详情，请参阅 setPaymentRequestEnabled() 的文档。
  • 推出了实验性 Navigation API，可在 WebView 中增强网页导航跟踪和管理功能。这些 API 可提供有关导航事件的详细信息。首先，使用新的 WebNaviagationClient 接口的实现调用 WebViewCompat.setNavigationClient，以获取有关网页导航的详细信息。如需详细了解导航生命周期，请参阅导航演示的生命周期。

版本 1.6.0

2023 年 1 月 25 日

发布了 androidx.webkit:webkit:1.6.0。版本 1.6.0 中包含这些提交内容。

自 1.5.0 以来的重要变更

  • 添加了 ProcessGlobalConfig API，让应用能够提供需要在加载 WebView 之前设定的配置设置，例如 WebView 数据目录后缀。在应用启动期间应尽早设置并应用该配置，以确保在执行此操作后，任何其他线程才能调用一种方法来将 WebView 加载到进程中。
  • 添加了 CookieManagerCompat API，用新的 getCookieInfo 方法扩展 android.webkit.CookieManager，该方法会检索特定网址上设置的所有 Cookie 的所有属性。这与 CookieManager 中现有的 getCookie API 不同，后者仅返回 Cookie 的名称和值属性。
  • WebSettingsCompat 添加了新方法来启用/停用 EnterpriseAuthenticationAppLinkPolicy 的效果（如果管理员已在 WebView 中设置）及获取当前设置。此功能可让 WebView 打开身份验证应用，而非打开身份验证网址，不会影响不受企业政策管理的设备。
  • 向 WebSettingsCompat 添加了一个新 API，可让应用将 X-Requested-With 标头中的应用软件包名称显式发送到已列入许可名单的来源。一直以来，系统都是在 WebView 发出的每个请求中发送该标头。

自 1.0.0 以来的重要变更

  • ProxyController API，用于为应用的 WebView 设置网络请求代理。
  • WebViewAssetLoader API，用于简化通过请求拦截从应用数据目录中加载 APK assets、resources 和文件的过程。这样可以在不停用 CORS 的情况下访问网络和本地资源。

  • RenderProcess API，用于管理 WebView 渲染程序服务，并检测行为不当的内容何时会导致 WebView 渲染程序无响应。
  • 更新了现有 API，加入了可为 null 性（@NonNull、@Nullable）和线程（@UiThread、@WorkerThread）注解。
