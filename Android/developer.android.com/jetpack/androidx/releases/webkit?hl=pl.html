Deklarowanie zależności

Aby dodać zależność od Webkit, musisz dodać repozytorium Google Maven do projektu. Więcej informacji znajdziesz w repozytorium Maven Google.

Dodaj zależności dotyczące potrzebnych artefaktów w pliku build.gradle aplikacji lub modułu:

Odlotowe

dependencies {
    implementation "androidx.webkit:webkit:1.14.0"
}

Kotlin

dependencies {
    implementation("androidx.webkit:webkit:1.14.0")
}

Więcej informacji o zależnościach znajdziesz w artykule Dodawanie zależności kompilacji.

Opinia

Twoja opinia pomoże nam ulepszyć Jetpacka. Jeśli odkryjesz nowe problemy lub masz pomysły na ulepszenie tej biblioteki, daj nam znać. Zanim utworzysz nową kartę, zapoznaj się z dotychczasowymi problemami w tej bibliotece. Możesz dodać swój głos do istniejącego problemu, klikając przycisk gwiazdki.

Wersja 1.15

Wersja 1.15.0-alpha02

27 sierpnia 2025 roku

Publikacja androidx.webkit:webkit:1.15.0-alpha02 Wersja 1.15.0-alpha02 zawiera te zmiany.

Nowe funkcje

  • Dodaje Profile#preconnect, które umożliwia aplikacjom otwieranie połączenia ze źródłem przed przejściem do niego, aby przyspieszyć przyszłe wczytywanie. (I8db41)
  • Do WebViewCompat dodaliśmy nowy interfejs API prerenderUrlAsync, który umożliwia aplikacjom spekulacyjne wstępne renderowanie adresów URL przed ich wyświetleniem w komponencie WebView.
    Wstępne renderowanie może znacznie poprawić szybkość wczytywania, ponieważ strona jest pobierana z wyprzedzeniem. Gdy użytkownik przejdzie do adresu URL, wstępnie wyrenderowana strona zostanie wyświetlona natychmiast.
     Te interfejsy API zostały wcześniej udostępnione w wersji eksperymentalnej, a teraz można ich używać bez adnotacji @OptIn. (Ie14f5)
  • Podczas wywoływania interfejsu startUpWebView API możesz teraz określić zestaw profili, które mają być wczytywane synchronicznie. Konfiguruje się to przez przekazanie nazw profili do nowej metody w WebViewStartUpConfig.Builder (I9a6e6).
  • Dodaj nowy interfejs API do WebSettingsCompat, aby włączyć lub wyłączyć BackForwardCache w obiekcie WebSettings. Ten interfejs API został wcześniej udostępniony w wersji eksperymentalnej, a teraz można go używać bez adnotacji @OptIn. (Ie413b)
  • Dodano obsługę zastępowania formatów User-Agent. Klasa UserAgentMetadata ma nowe metody pobierania i ustawiania formatów metadanych User-Agent. UserAgentMetadata.Builder ma nową metodę setFormFactors() zastępowania Sec-CH-UA-Form-Factors wskazówki dotyczącej klienta użytkownika. Bieżące wartości zastąpienia można pobrać za pomocą nowej metody getFormFactors(). (I6d08c, b/430554841)

Zmiany w interfejsie API

  • Dodaj interfejs API do ProcessGlobalConfig, aby skonfigurować inicjowanie wątku interfejsu WebView (I5e85a, b/433273850, b/397372092).
  • Dodawanie nowej nazwy funkcji dla interfejsu API WebViewStartupConfig.Builder#setProfilesToLoadDuringStartup (I36451)
  • Przeniesienie domyślnego minSdk z API 21 na API 23 (Ibdfca, b/380448311, b/435705964, b/435705223)
  • Zaktualizuj interfejs API nagłówka dopasowanego do źródła, aby umożliwić zastępowanie wartości nagłówka przez wielokrotne wywoływanie funkcji setOriginMatchedHeader dla tej samej nazwy nagłówka. Dodaje też nową metodę hasOriginMatchedHeader, która sprawdza, czy nazwa nagłówka jest już używana.
    Ta zmiana oznacza również, że interfejs API jest eksperymentalny, ponieważ obecnie obsługuje tylko ustawianie pojedynczej wartości i mapowania pochodzenia dla każdej wartości nagłówka. W przyszłości prawdopodobnie ulegnie zmianom, aby obsługiwać wiele wartości mapowanych na różne pochodzenia. (I03365, b/419469873, b/429269470)
  • Wymóg dotyczący wątku interfejsu użytkownika w przypadku Profile#prefetchUrlAsync() został usunięty. Można go teraz wywoływać z dowolnego wątku. (I83d57)
  • Interfejs OutcomeReceiverCompat został usunięty na rzecz oficjalnej wersji androidx.core.os.OutcomeReceiverCompat. Biblioteka wymaga teraz compileSdk w wersji 35 lub nowszej. (Ie1ea5)
  • Zmienia nazwę [is/set]IncludeCookiesOnShouldInterceptRequestEnabled na [are/set]CookiesIncludedInShouldInterceptRequest w WebSettingsCompat. (Ife9e9, b/428924452)
  • Dodanie adnotacji do eksperymentalnych interfejsów API w Profile.ExperimentalUrlPrefetch, które wcześniej zostały pominięte. (Ic54eb, b/417458061)

Wersja 1.15.0-alpha01

2 lipca 2025 r.

Publikacja androidx.webkit:webkit:1.15.0-alpha01 Wersja 1.15.0-alpha01 zawiera te zmiany.

Nowe funkcje

  • Nowy interfejs API do ustawiania par nazwa-wartość nagłówka HTTP w żądaniach kierowanych do określonych źródeł.
    Metoda Profile.setOriginMatchedHeader umożliwia aplikacji określenie, że nazwa i wartość nagłówka powinny być ustawiane we wszystkich żądaniach HTTP/HTTPS kierowanych do źródeł zgodnych z podaną listą dozwolonych, gdy są wysyłane z elementów WebView i procesów service worker, które używają danego profilu. Interfejs API umożliwia tylko jedno mapowanie dla każdej nazwy nagłówka i zgłasza wyjątek, jeśli nazwa nagłówka jest już ustawiona.
     Użyj Profile.clearOriginMatchedHeader, aby usunąć istniejące mapowanie, a Profile.clearAllOriginMatchedHeaders, aby usunąć wszystkie mapowania. (Ida7f1, b/419469873)
  • Nowy interfejs API umożliwiający dostęp do plików cookie i ich ustawianie w ramach shouldInterceptRequest wywołań zwrotnych w przypadku komponentu WebView i procesów Service Worker.

Wersja 1.3.0

19 sierpnia 2020 r.

Publikacja androidx.webkit:webkit:1.3.0 Wersja 1.3.0 zawiera te zmiany.

Główne zmiany od wersji 1.2.0

  • Interfejs ForceDarkStrategy API zapewnia większą kontrolę nad przyciemnianiem WebView (przyciemnianie treści CSS/webowych w porównaniu z automatycznym przyciemnianiem).
  • Interfejs WebMessageListener i powiązane z nim interfejsy API zapewniają prosty i bezpieczny mechanizm nawiązywania komunikacji między treściami internetowymi a aplikacją osadzającą WebView.
  • isMultiProcessEnabled Interfejs API do sprawdzania, czy WebView działa w wielu procesach. Jest to możliwe od Androida O. Oznacza to, że treści internetowe są renderowane w procesie renderowania w piaskownicy, który jest oddzielony od procesu aplikacji. Ten mechanizm renderowania w piaskownicy może być udostępniany innym elementom WebView w tej samej aplikacji, ale nie jest udostępniany innym procesom aplikacji.

Wersja 1.1.0

7 listopada 2019 r.

Publikacja androidx.webkit:webkit:1.1.0 Wersja 1.1.0 zawiera te zmiany.

Ważne zmiany od wersji 1.0.0

  • Interfejsy API typu getter uzupełniające funkcje setWebViewClient() i setWebChromeClient().
  • interfejsu ProxyController API do ustawiania serwera proxy żądania sieciowego dla widoków WebView aplikacji;
  • Interfejs WebViewAssetLoader API, który upraszcza wczytywanie zasobów APK, zasobów i plików z katalogu danych aplikacji przez przechwytywanie żądań. Umożliwia to dostęp do zasobów internetowych i lokalnych bez wyłączania CORS.

  • Interfejsy RenderProcess API do zarządzania usługami renderowania WebView i wykrywania, kiedy nieprawidłowo działające treści powodują, że renderowanie WebView przestaje odpowiadać.
  • Zaktualizowano istniejące interfejsy API, aby uwzględnić adnotacje dotyczące możliwości przyjmowania wartości null (@NonNull, @Nullable) i wątków (@UiThread, @WorkerThread).

Wersja 1.1.0-beta01

5 września 2019 r.

Publikacja androidx.webkit:webkit:1.1.0-beta01 Zmiany zawarte w tej wersji znajdziesz tutaj.

Nowe funkcje

  • Nowa funkcja InternalStoragePathHandler w WebViewAssetLoader do wczytywania plików z katalogu danych aplikacji.

Zmiany w interfejsie API

  • ProxyConfig#getProxyRules() zwraca teraz niemodyfikowalną listę instancji ProxyRule, która jest nową klasą przechowującą filtr schematu i odpowiadający mu adres URL serwera proxy.

Poprawki błędów

  • WebViewAssetLoader domyślnie ustawia typ MIME „text/plain” (zamiast wartości null), gdy nie może odgadnąć typu MIME na podstawie ścieżki pliku.
  • WebViewAssetLoader nie zgłasza już NullPointerException podczas wczytywania plików ze znakami specjalnymi w nazwach ścieżek.

  • WebViewAssetLoader udostępnia interfejs PathHandler, który umożliwia aplikacjom tworzenie niestandardowych funkcji obsługi ścieżek.

  • Zaktualizowano istniejące interfejsy API, aby uwzględnić adnotacje dotyczące możliwości przyjmowania wartości null (@NonNull, @Nullable) i wątków (@UiThread, @WorkerThread).
